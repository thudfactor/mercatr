# Feature Spec: Genre Diversity Constraints & `--expand` Flag

## Problem

Playlist outputs show a gravitational pull toward indie, alternative, and hip-hop — genres overrepresented in music criticism and LLM training data. Metal, blues, jazz, country, classical, electronic, and non-Western traditions are systematically underrepresented even when thematically appropriate. The existing "genuine diversity" instruction in the user prompts is not strong enough to counteract this pull.

---

## Solution Overview

Two-layer approach:

1. **Baseline constraint** added to all prompt templates — counteracts genre bias by default, without changing the user experience
2. **`--expand` flag** on the CLI — shifts to a more aggressive diversity posture for users who want it, expressed in plain language with a noticeable effect

---

## Part 1: Baseline Prompt Constraints

### What changes

A **Genre Diversity Check** block is added to the system prompt of all three query types (`explore`, `bridge`, `theme`). It replaces the vague "all genres, decades, and global traditions" in the existing persona line with a concrete self-audit requirement.

The block is added **after** the Last.fm data section and **before** the user prompt in each template. It fires on every query.

---

### Block: Genre Diversity Check (baseline)

```
## Genre Diversity Check

Before finalizing your response, audit your selections against the following:

**Overrepresented by default — actively resist defaulting to these:**
- Indie / indie rock / indie folk
- Alternative rock
- Singer-songwriter (acoustic, confessional)
- Hip-hop / rap (mainstream US)

**Underrepresented by default — actively seek these when thematically justified:**
- Blues (Delta, Chicago, electric)
- Jazz (any era or tradition)
- Metal (any subgenre)
- Country / Americana / bluegrass
- Classical / contemporary classical / minimalism
- Electronic / ambient / techno (non-indie)
- Soul / R&B / gospel
- Reggae / dub
- Non-Western traditions (afrobeat, cumbia, qawwali, etc.)
- Pre-1960s popular music

**The test:** Could your selections have been generated by a recommendation algorithm trained on Pitchfork and Rolling Stone? If yes, reconsider at least two choices.

This is not a quota system. Every selection must genuinely serve the theme. But when two songs are equally thematically valid, prefer the one from an underrepresented genre or decade.
```

---

### Per-template placement

**`explore.md` system prompt** — insert after `## Last.fm Data` section, before the response instructions.

**`bridge.md` system prompt** — insert after `## Overlapping Tags` section, before the response instructions.

**`theme.md` system prompt** — insert after `## Top Tracks Tagged` section, before the response instructions.

## Part 2: `--expand` Flag

### User-facing behavior

```bash
mercatr explore "elliott smith"
mercatr explore "elliott smith" --expand
```

Without `--expand`: baseline constraints apply (as above).

With `--expand`: the Genre Diversity Check block is replaced with a more aggressive version, and the user prompt requirements section is also modified.

The flag is available on all three query types.

---

### Block: Genre Diversity Check (`--expand` variant)

Replaces the baseline block entirely when `--expand` is active:

```
## Genre Diversity Check (Expanded Mode)

You are operating in expanded discovery mode. The goal is to surface music 
a listener would not find through any standard recommendation system.

**Hard constraints for this mode:**
- No more than 2 selections from indie, alternative, or mainstream hip-hop combined
- At least 1 selection predating 1970
- At least 1 selection from a non-anglophone tradition
- At least 1 selection from metal, blues, jazz, or classical

**Your posture:** Treat the obvious choices as placeholders. Make them, then 
replace them with something that serves the same thematic function from a 
genre or decade you haven't touched yet.

The thematic connections must still hold. Diversity without justification is 
just randomness. But if you find yourself writing "this indie artist captures 
the same feeling as..." more than once, stop and find a different path.
```

---

### User prompt additions (`--expand` variant)

In the user prompt's Requirements section, add one line after the existing diversity requirement:

```
- **Expanded mode is active**: Treat genre and decade diversity as a hard 
  requirement, not a preference. If a selection is from indie, alternative, 
  or mainstream hip-hop, it must earn its place against stronger competition 
  from underrepresented genres.
```

---

## CLI Integration

### Flag parsing

`--expand` is a boolean flag, no argument. Default is `false`.

```
mercatr theme "loneliness in crowded places" --expand
mercatr bridge "nick cave" "sabrina carpenter" --expand
mercatr explore "elliott smith" --expand
```

### Prompt construction logic

```
if (expand) {
  systemPrompt = buildSystemPrompt(queryType, { diversityBlock: 'expand' })
  userPrompt = buildUserPrompt(queryType, { expandMode: true })
} else {
  systemPrompt = buildSystemPrompt(queryType, { diversityBlock: 'baseline' })
  userPrompt = buildUserPrompt(queryType, { expandMode: false })
}
```

The diversity block should be a named slot in the template rather than hardcoded — both variants live in the template file, selected at build time.

---

## Logging

Add `expandMode` boolean to the top-level log entry:

```json
{
  "timestamp": "...",
  "queryType": "theme",
  "expandMode": true,
  "preflight": [...],
  "templatePath": "...",
  "model": "...",
  "systemPrompt": "...",
  "userPrompt": "...",
  "response": "..."
}
```

This lets you compare outputs across modes in analysis without reading the full prompt.

## Acceptance Criteria

- [ ] Baseline Genre Diversity Check block is present in system prompts for `explore`, `bridge`, and `theme`
- [ ] `--expand` flag is available on all three query types
- [ ] `--expand` replaces the baseline diversity block with the expanded variant
- [ ] `--expand` adds the expanded mode line to the user prompt requirements
- [ ] `expandMode` boolean is logged in all query outputs (including `false` for baseline runs)
- [ ] Without `--expand`, no user-visible change to existing flow
- [ ] Both diversity block variants live in template files, not hardcoded in CLI logic

---

## Notes for Tuning

The hard constraints in `--expand` mode ("no more than 2 indie/alternative selections") are starting points, not final values. After a few runs, check whether the constraint is producing genuinely surprising results or just forcing awkward inclusions. The Pitchfork/Rolling Stone test in the baseline block is intentionally informal — it gives the model a cultural reference point for self-auditing rather than a mechanical rule.

If baseline mode still trends too heavily toward the prestige canon after a few runs, tighten the baseline before expanding the `--expand` constraints further.
